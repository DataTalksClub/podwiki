## Definition and Scope

Documentation is the written and recorded material that helps people use,
maintain, explain, or evaluate technical work. In the archive, documentation
includes READMEs, quickstarts, and guides. API references and examples also
matter. It also includes contribution guides and design docs.

Decision logs and tracking plans recur too. Model cards and datasheets belong
here as well, and runbooks, demos, and portfolio repo tours also fit.

Use this page for documentation as a practice across tools, teams, and open
source. Use [Technical Writing]({{ '/wiki/technical-writing/' | relative_url }})
for the writer's workflow and [Developer Relations]({{ '/wiki/developer-relations/' | relative_url }})
for education, demos, and adoption work around developer tools.


## Recurring Archive Themes

Documentation makes tools usable.

Guests repeatedly say that good code isn't enough. A project needs to tell
people what the tool does, how to install it, and what problem it solves. It
also needs examples and a clear way to ask for help or contribute.

Documentation preserves decisions.

Eugene Yan expands documentation beyond public blog posts. He discusses design
docs, working-backwards documents, decision logs, and rationales as team memory.
This kind of documentation helps future teammates understand not only what was
built, but why.

Documentation is a contribution path.

Open-source episodes treat docs fixes and examples as legitimate
contributions. A reproducible issue, clearer error message, quickstart, or
guide can reduce maintainer load and help users more than a feature PR.

Documentation is part of production practice.

Production and data-platform episodes connect documentation to ownership,
monitoring, governance, and stakeholder alignment. They also connect it to
recovery. The archive uses documentation as a reliability practice, not only a
learning resource.

## Documentation Types

These documentation types recur across wiki topics.

- Adoption docs: README, quickstart, installation notes, guide, tutorial, API
  reference, and examples.
- Contribution docs: contribution guide, issue template, PR checklist,
  maintainer expectations, test commands, and coding style.
- Team memory: design doc, decision log, rationale, runbook, model card,
  datasheet, and tracking plan.
- Portfolio docs: repo tour, problem statement, setup steps, screenshots,
  model or pipeline limitations, and verification notes.
- DevRel material: demos, videos, talks, notebooks, blog posts, and workshop
  material that make a tool easier to try.
