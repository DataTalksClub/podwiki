QA to ML and data engineering moves from testing and verification toward
building data systems, models, and production workflows. QA experience helps
because the person already knows checklists, edge cases, and failure reports.
They also know regression thinking and the discipline of proving that a system
works.

The gap isn't only technical. In the main QA transition episode, the candidate
also had to present projects and update the CV. Interview preparation and
confidence mattered too.


## Transition Profile

This path fits QA engineers, testers, field testers, and release testers.
Technical support people can fit too. They usually bring experience with defects,
checklists, repeated validation, or hardware/software release cycles. It's
especially strong when the person already has programming fundamentals or enough
IT background to move into Python, SQL, cloud services, and pipelines.

The archive's main example is Alvaro Navas Peire. He moved from testing Android
phone prototypes and field behavior into ML, data engineering, and NLP project
management. He used structured courses, public notes, and portfolio projects.
Cloud practice and interview coaching also mattered.

## Transferable Skills

QA work gives candidates a testing vocabulary they can reuse in data work.

- Test design: QA work maps to validation, edge cases, acceptance criteria, and
  data quality checks.
- Reproducibility: testers know that a bug report needs steps and context. It
  also needs expected results.
- Attention to detail: QA habits help with schema changes, nulls, duplicates,
  metric drift, and model evaluation.
- Release judgment: testing work builds respect for regression risk and
  production readiness.
- Communication: QA reports train people to explain a failure to engineers,
  managers, and vendors.

## Missing Skills

The target role decides which gaps matter most.

For machine learning:

- Python for data work, notebooks, scripts, and model training
- statistics, evaluation metrics, leakage, validation, and feature work
- model deployment basics and enough cloud knowledge to run projects
- role-specific depth such as NLP, computer vision, or time series

For data engineering:

- SQL, data modeling, batch pipelines, orchestration, and warehousing
- cloud storage, compute, permissions, and cost awareness
- Docker, Git, CI, and reproducible project setup
- data quality checks that go beyond manual QA checklists

For both:

- interview framing, project narration, and a CV that makes the prior QA work
  relevant to the target role

## Portfolio Evidence

The QA transition portfolio should show both building and testing.

Good projects include:

- an end-to-end ML project with data validation, training, deployment, and a
  short evaluation report
- a data pipeline that ingests raw data, validates assumptions, writes modeled
  tables, and records data quality failures
- a cloud-deployed project using one provider deeply enough to explain storage,
  compute, permissions, and cost tradeoffs
- public technical notes that document setup, errors, fixes, screenshots, and
  reasoning
- a role-specific project such as NLP classification, image classification, or
  time-series forecasting, with an honest explanation of why the metric fits

The archive gives a useful presentation rule: describe the dataset and task
factually. Include the method and result. Don't apologize for the project before
the interviewer has evaluated it.

## Episode Evidence

These episodes define the QA transition evidence.

- [Transition from QA to Machine Learning and Data Engineering](https://datatalks.club/podcast.html):
  At 3:41, the prior QA work includes phone prototypes, field tests, and
  checklists. CTS tests, RF constraints, and repeated firmware validation also
  matter. At 8:35, the pivot starts after leaving QA. At 13:32, the structured
  path includes postgraduate study and Neuromatch. Machine Learning Zoomcamp and
  Data Engineering Zoomcamp add project practice.

  At 17:57, job search work includes soft skills, coaching, and CV redesign. At
  24:57, Zoomcamp projects become portfolio evidence. At 28:52, Alvaro learns
  to present projects objectively instead of underselling them.
- [Machine Learning Portfolio Projects](https://datatalks.club/podcast.html):
  The archive repeatedly values projects that show data work, modeling,
  deployment, and communication rather than course completion alone.
- [Data Engineering Portfolio Projects](https://datatalks.club/podcast.html):
  Data engineering evidence gets stronger when projects include ingestion,
  orchestration, storage, and transformations. Quality checks and a clear README
  make the evidence stronger.
- [Hiring Data Scientists and Analysts](https://datatalks.club/podcast.html):
  Hiring episodes support the same lesson: CVs and interviews need problem
  evidence, not tool lists.
