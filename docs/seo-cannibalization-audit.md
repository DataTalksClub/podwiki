# Podwiki ↔ Main-site SEO cannibalization audit

_Audit date: 2026-07-02. Read-only. Both sites live on `datatalks.club` (main = `/`, podwiki = `/podwiki/`), so two pages targeting one query = cannibalization._

## 1. Method + the score-vs-verdict caveat

I ran `scripts/find_duplicates.py` and worked its "CROSS-SITE overlap" ranking (BM25 of each of the 275 wiki pages vs its nearest main-site `_posts` article). I evaluated the full defensible set: all 63 pages carrying a `keyword:` field plus every non-keyword page scoring above ~45, and batch-judged the tail. For each I opened the podwiki front matter (title / keyword / summary / search_intent / tags) and the top main-site match's front matter + intro, then judged shared **query/intent**, not shared vocabulary.

**A high BM25 score is a signal, not a verdict.** The single biggest source of false positives is shared domain vocabulary: podwiki **concept hubs** (e.g. `mlops.md`, `retrieval-augmented-generation.md`, `llmops.md`) and **comparison pages** (`etl-vs-elt`, `graph-rag-vs-vector-rag`) score highly against main-site **branded course pages** (MLOps/LLM/DE Zoomcamp, "Free DataTalks.Club Courses") purely because they reuse the same terms — but a concept query ("what is mlops") and a branded course query ("mlops zoomcamp") are *different queries with different owners*. Per the site rule (concept → podwiki wiki topic; comparison/decision → podwiki; branded/course → main), those are **OK**, not cannibalizing. The real cannibalization is the small set where a main-site article actually targets the *same* head query as a podwiki page — almost always where a retired-Ubersuggest `keyword:` leftover mirrors a specific main-site editorial/opinion/course article. `airflow-docker-compose` (already being removed) is the reference case: a podwiki how-to duplicating the main site's "How to Setup a Lightweight Local Version for Airflow" how-to on the same procedural query.

## 3. Clusters by main-site owner

- **"What is DataOps exactly?" + "DataOps: Similarities and Differences…"** — a 7-page podwiki DataOps sprawl orbits these two articles: `dataops` (#1) and `dataops-vs-data-engineering` (#2) are direct collisions; `dataops-tools`, `dataops-platforms`, `dataops-engineer-role`, `dataops-checks-for-data-pipelines`, `mlops-vs-dataops` are borderline sub-intents (see §4). The high scores (76–101) are inflated by the dense shared DataOps vocabulary. Biggest cluster; consolidating the DataOps pages into one concept hub + one comparison would fix both the main-site collision and the podwiki-internal sprawl.
- **"DevOps vs MLOps…"** — `mlops-vs-devops` (#3, direct). `model-monitoring`, `machine-learning-tools`, `bioinformatics-data-science`, `cdc` also map here but are pure vocabulary false positives (OK).
- **"How Do Professionals Use AI Tools for Personal Productivity?"** — `ai-tools-for-personal-productivity` (#4, direct); `solopreneur-data-scientist` maps here but is a different query (OK).
- **DE course pages ("15 Free DE Courses" / "DE Zoomcamp")** — `data-engineering-courses` (#5, direct); `data-engineering-certification` borderline; roadmap/portfolio/tool pages mapping here are OK.
- **False-positive magnets (no cannibalization):** the branded course pages (**MLOps Zoomcamp, LLM Zoomcamp, ML Zoomcamp, AI Dev Tools Zoomcamp, "Free DataTalks.Club Courses"**) are the nearest neighbour for ~120 wiki concept/roadmap/transition pages purely on shared terminology — all OK (concept ≠ branded query). Likewise **"How I Landed a Job As a Product Analyst"** (a personal job-hunt narrative) is nearest neighbour for the whole product/analyst cluster (`product-analyst`, `product-analyst-vs-data-analyst`, `data-analyst-role`, etc.) but owns a *different* query (personal story, not role definition) — OK. Same for **"Data Science Manager vs Expert"** vs podwiki's *other* role comparisons, and **"Open Source and Free AI Agent Evaluation Tools"** (a tool listicle) vs podwiki's agent *concept* hubs.

## 5. Counts

- **CANNIBALIZING: 5** — `dataops`, `dataops-vs-data-engineering`, `mlops-vs-devops`, `ai-tools-for-personal-productivity`, `data-engineering-courses`. **All 5 carry a `keyword:` field.** (Plus `airflow-docker-compose`, already being removed — reference case, not counted.)

- **OK: ~65** — high score but genuinely different intent (concept hub vs branded course; podwiki-owned comparison/roadmap/transition; personal-story main-site match). Includes the ~120-page long tail whose nearest neighbour is a Zoomcamp/course page or the product-analyst personal story.
- **Keyword-field pages total: 63** (62 excluding the already-removed `airflow-docker-compose`). The overwhelming majority of `keyword:` pages are NOT cannibalizing the main site — they target comparison/concept/roadmap queries the main site does not own — but they remain the highest-risk pool and every confirmed cannibalizing page came from it.
